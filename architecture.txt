# Архитектура проекта DIGITAL AGORA

## Общая структура

Проект представляет собой децентрализованную платформу управления ТСЖ с использованием блокчейна, ИИ и веб-интерфейса.

├── backend/               # Node.js сервер (Express, JWT, bcrypt, файловое хранилище)
│   ├── models/            # Модель пользователя (UserFileModel.js)
│   ├── routes/            # Маршруты аутентификации (auth.js)
│   ├── middleware/        # Проверка JWT (auth.js)
│   ├── server.js          # Точка входа бэкенда
│   ├── .env               # Переменные окружения (порт, JWT_SECRET)
│   └── package.json       # Зависимости бэкенда
├── frontend/              # Статический фронтенд (HTML/CSS/JS)
│   └── public/            # Публичная папка (index.html – единая страница)
├── contracts/             # Смарт-контракты (Hardhat, Solidity)
│   ├── AgoraToken.sol     # ERC20 токен для голосования
│   ├── AgoraGovernor.sol  # Управление предложениями и голосование
│   └── Timelock.sol       # Отложенное выполнение (опционально)
├── scripts/               # Скрипты развёртывания (deploy.js)
├── test/                  # Тесты смарт-контрактов
├── artifacts/             # Скомпилированные контракты (ABI)
├── cache/                 # Кеш Hardhat
├── data/                  # Файловое хранилище (users.json, proposals.json) – автоматически создаётся
├── deployed.json          # Адреса развёрнутых контрактов
├── hardhat.config.js      # Конфигурация Hardhat
├── package.json           # Зависимости корня (Hardhat, ethers)
└── README.md              # Документация проекта

## Компоненты

### Бэкенд (Node.js + Express)
- **Назначение**: аутентификация пользователей, хранение данных (JSON-файлы), выдача JWT.
- **Порт**: 5000 (настраивается через .env)
- **API**:
  - `POST /api/auth/register` – регистрация (хеширование bcrypt)
  - `POST /api/auth/login` – вход, возврат JWT
- **Хранилище**: `data/users.json` – массив пользователей с полями: lastName, firstName, patronymic, birthDate, residence, walletAddress, email, password (хеш).

### Фронтенд (статический HTML/CSS/JS)
- **Единая страница**: `frontend/public/index.html`
- **Менеджер состояния**: localStorage (токен, пользователь, предложения, баланс)
- **Основные разделы**:
  - Регистрация / вход
  - Предложения (создание, голосование через смарт-контракт)
  - ИИ-анализ (демо)
  - Подрядчики (демо-поиск)
  - Профиль (данные пользователя, баланс токенов)
- **Подключение к блокчейну**: через MetaMask, ethers.js (CDN)
- **Стили**: встроенный CSS (адаптивный, тёмно-синяя тема)

### Блокчейн (Hardhat, локальная сеть)
- **Сеть**: `localhost:8545`
- **Контракты**:
  - `AgoraToken` – ERC20 с функцией `getVotes` (для голосования)
  - `AgoraGovernor` – управление предложениями (propose, castVote)
  - `Timelock` – опциональная задержка выполнения
- **Развёртывание**: `npx hardhat run scripts/deploy.js --network localhost`
- **ABI и адреса** хранятся в `artifacts/` и `deployed.json`.

## Взаимодействие компонентов

1. Пользователь открывает `http://localhost:3000` (статический сервер).
2. Регистрируется через бэкенд (API). Бэкенд сохраняет данные в `users.json`, возвращает JWT.
3. После входа фронтенд сохраняет токен и отображает дашборд.
4. Пользователь подключает MetaMask к локальной ноде Hardhat.
5. Создание предложения: фронтенд вызывает `governor.propose()`.
6. Голосование: `governor.castVote(proposalId, support)`.
7. Баланс токенов считывается через `token.balanceOf()`.
8. ИИ и подрядчики – демо-функции (в будущем замена на реальные API).

## Порты и серверы

- **Бэкенд**: `http://localhost:5000`
- **Блокчейн RPC**: `http://localhost:8545`
- **Фронтенд (статика)**: `http://localhost:3000` (раздаётся через `http-server` или `serve`)

## Требования к окружению

- Node.js 18.x или выше
- npm 9.x
- MetaMask (расширение браузера)
- Hardhat (локальный запуск)
- Для продакшена: замена `data/` на реальную БД (MongoDB/PostgreSQL), хостинг фронтенда и бэкенда.

## Безопасность

- Пароли хешируются bcrypt.
- JWT подписывается секретом из `.env`.
- CORS настроен для `localhost:3000` и `127.0.0.1:3000`.
- Хранение токена в `localStorage` – удобно для разработки; в продакшене рекомендуется `httpOnly cookie`.
